## What Changed Since v0.1.0-rc.2

This candidate addresses the three sync-plane issues rc.2 shipped as known
limitations (#112 — its history disclosure, with one tracked residual — plus
#113 and #114, both closed):

- **Join-bootstrap privacy gate (issue #112, PRs #117 and #120).** Since
  rc.2, `WantMembership` serves the causal closure of the authorization class,
  which can carry chat that entered the membership ancestry — and while a join
  window (`room tail --accept-joins`) was open, any provisionally-admitted
  unknown device could pull that chat with no invite. A provisional peer must
  now prove invite possession (`ProveCapability { room_id, invite_id,
  capability_secret }`, verified against the on-log invite's
  `capability_hash`) before the closure is served; the proof presentation is
  automatic in `room join`, `Node::spawn_join_bootstrap`, and the SDK
  examples. This is a bootstrap privacy gate only — the convergent `gate_join`
  remains the unchanged authorization authority. Residual: outbound live
  fan-out to a still-connected unproven provisional dialer is not yet gated
  (issue #121, see Known Limitations).
- **Membership-sync room-size ceiling removed (issue #113, PR #118).** The
  `WantMembership` requester claimed every held event id, so rooms near ~30k
  events exceeded the 1 MiB frame cap and membership anti-entropy silently
  stalled. The claim is now a bounded ancestry sample (placed DAG heads +
  recent-lamport slab + per-tick rotating window, ~17 KiB regardless of room
  size) and the responder subtracts each claimed id plus every stored ancestor
  of it. `Events` responses are byte-budgeted into under-cap chunks (a
  512-frame batch of large message bodies could exceed the frame cap and stall
  a bootstrap at any room size), locally-authored frames too large to ever
  deliver are refused at publish, and the deterministic sim now enforces the
  frame cap so this failure class stays visible to Gate-D tests.
- **Deep pure-chat gap heal (issue #114, PR #116).** A member returning across
  a >64-deep linear pure-chat gap accepted no new chat: the backfill chase
  re-requested already-parked parents, the tick retry was a silent no-op for
  parked chains, and the 64-deep depth/park caps evicted the middle of
  legitimate chains. Backfill now skips in-flight parents, the tick retry
  drives from recorded missing sets, and the depth and per-author park bounds
  are raised to 1024 (still finite — the Gate-D bounded-backfill requirement
  holds, widened).

See `crates/iroh-rooms/CHANGELOG.md` for the full entries.

## Upgrade Note — Do Not Mix rc.2 And rc.3

Mixed rc.2/rc.3 rooms break in **both** directions:

- An rc.2 **joiner** never sends the capability proof an rc.3 admin now
  requires, so its `room join` receives no provisional bootstrap and times
  out.
- An rc.3 **joiner** bootstrapping against an rc.2 responder hard-stalls once
  the joiner holds more than ~1k events (the rc.2 responder subtracts the new
  bounded claim as an exact id set).
- An rc.2 requester whose store carries a hole from a swallowed insert error
  (issue #119) claims the unplaced rows above it, so an rc.3 responder never
  re-serves the missing ancestor that rc.2-to-rc.2 subtraction would have
  healed.

**Every room member — especially the room admin — must upgrade to rc.3.**

## Supported Scope

This candidate supports:

- Identity creation.
- Private room creation.
- Key-bound room invites and joins — including joining a room after the
  conversation has started, now gated on proof of invite possession.
- Signed room messages.
- Verified file/artifact sharing.
- Authenticated loopback-only live TCP pipes with explicit per-member allow
  lists.
- Explicitly invited agent participants and signed `agent.status` updates.
- Returning-member catch-up across pure-chat gaps up to the 1024-deep
  backfill bound.
- Membership reconciliation independent of room size (no ~30k-event ceiling).
- Best-effort local audit trail at `<IROH_ROOMS_HOME>/audit.ndjson`.

## Known Limitations

Read before installing or running this candidate:

- No central application server.
- No guaranteed offline message delivery.
- Local storage is plaintext.
- Invite tickets are password-grade capabilities.
- No native ticket-specific invite revocation; ADR-0002 accepts a bounded
  leaked-ticket model for beta scope only.
- A pure-chat gap deeper than the (finite) 1024-event backfill bound still
  does not fully heal: the chase is bounded by design and membership still
  converges (issue #114 residual, tracked under #102).
- An adversarially wide DAG (thousands of concurrent branches) can hold a
  peer's membership pull in bounded per-tick duplicate re-serves until the
  claim's rotating window anchors; the full answer is the deferred Meyer range
  reconciliation (tracked under #102).
- While a join window (`room tail --accept-joins`) is open, a
  provisionally-admitted dialer that has not proven invite possession no
  longer receives any room history — but events published **while it stays
  connected** still fan out to it (issue #121). The window is
  operator-controlled; close it when not expecting joins.
- A store hole left by a swallowed insert error heals from peers on re-serve,
  but a hole in a region no peer serves does not, and the fold/store can
  disagree until restart (issue #119).
- An oversized event that entered an rc.2 log before the publish-size guard
  existed remains undeliverable to peers (re-served and dropped, now logged).
- `audit.ndjson` is local best-effort audit, not remote, centrally retained,
  tamper-evident, or compliance-grade audit.
- Binary artifacts are checksummed but not project-signed.
- SDK crate remains `publish = false`; SDK use is source/workspace-based for
  this candidate.
- Gate A evidence carries over from the 2026-07-07 refresh (no transport-layer
  performance change in this candidate); the cellular relay-throughput caveat
  and missing home-NAT-to-CGNAT reverse-leg waivers remain in force.

## Security And Privacy Notes

- Storage posture: ADR-0001 scoped plaintext beta on trusted local machines.
- Invite posture: ADR-0002 bounded leaked-ticket model; no native invite
  revocation in this candidate.
- Audit posture: ADR-0003 local `audit.ndjson`; no central retention,
  tamper-evidence, or compliance claim.
- Narrowed in this candidate: the rc.2 provisional join-bootstrap disclosure
  bound (issue #112) — the membership closure (room **history**) is now served
  to a provisional dialer only after a verified capability proof. The proof
  carries the same secret the join later places on the log, over the
  authenticated transport link to the admin who minted it, so it discloses
  nothing new. Two tracked residuals: live fan-out to a still-connected
  unproven provisional dialer is not yet gated (issue #121), and proof
  accept/reject outcomes surface only through the tracing audit hooks — the
  CLI's `audit.ndjson` does not yet record them (issue #122).
- Security reporting path: use the private maintainer channel for secrets,
  data loss, authorization bypass, or remote code execution concerns. Public
  bug reports must follow `.github/ISSUE_TEMPLATE/bug_report.md` and must not
  attach full tickets, `identity.secret`, `rooms.db`, blob contents,
  unredacted `audit.ndjson`, or full data-directory backups.

  control message, not a room event).
- SQLite `user_version`: current schema v2 (unchanged from rc.2); v1 fixture

## Install

See `docs/operations/install-uninstall.md`.

Minimum binary install, with `TRIPLE` set to your platform:

```bash
TRIPLE=x86_64-apple-darwin  # or aarch64-apple-darwin, {x86_64,aarch64}-unknown-linux-gnu
shasum -a 256 -c "iroh-rooms-0.1.0-rc.3-$TRIPLE-71fbb5007bef.tar.gz.sha256"
tar -xzf "iroh-rooms-0.1.0-rc.3-$TRIPLE-71fbb5007bef.tar.gz"
install -m 0755 "iroh-rooms-0.1.0-rc.3-$TRIPLE-71fbb5007bef/iroh-rooms" \
  "$HOME/.local/bin/iroh-rooms"
iroh-rooms --version
```

On Linux, `sha256sum -c` replaces `shasum -a 256 -c`.

## Uninstall

```bash
rm -f "$HOME/.local/bin/iroh-rooms"
```

To remove local data, follow `docs/operations/install-uninstall.md`; do not
delete local data unless you accept losing local room state and identity
secrets.

## Rollback

Rollback requires a pre-upgrade data backup if data was opened by this
candidate. See `docs/operations/install-uninstall.md`. Rolling back to rc.2
reintroduces the provisional-bootstrap chat disclosure (issue #112), the
~30k-event membership-sync ceiling (issue #113), and the deep chat-gap wedge
(issue #114).
