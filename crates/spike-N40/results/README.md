# `spike-N40` results — committed run artifacts

This directory holds the durable, machine-readable outputs of the #145
N=40 measurement harness (spec `specs/measure-n40-v1-guardrails.md`).
Unlike `spike-nat`'s Gate A (a manual two-host run CI cannot prove), every
scenario here runs on deterministic loopback — CI proves the harness builds
and the loopback harness works at N=5 (`tests/self_check.rs`); the full
N=5/10/20/40 matrix is a manual run committed here.

## Layout

- `results.md` — the rolled-up matrix table, produced by
  [`report::results_md`](../src/report.rs) and pasted verbatim into
  `NOTES.md`'s decision memo. This is the canonical run artifact committed
  here; it is regenerated by the `n40-probe matrix` command below.
- `n40-probe matrix --json` / `n40-probe rebind --json` /
  `n40-probe sweep --json` emit one structured JSON document per run to the
  path passed via `--json`. The JSON schema is the union of the spec §7.1
  top-level fields (`crate_version`, `binary`, `iroh_version`, `os`, `arch`,
  `net_mode`, `relay`, `caveats`, `results`) and the per-scenario
  `ScenarioResult` records defined in
  [`report`](../src/report.rs). No `--json` output is committed by default;
  operators copy the dated JSON next to `results.md` when a run is executed.

## How to regenerate

```
cargo run -p spike-n40 --bin n40-probe -- self-check
cargo run -p spike-n40 --bin n40-probe -- matrix \
  --markdown crates/spike-N40/results/results.md \
  --json    crates/spike-N40/results/$(date +%F)-matrix.json
cargo run -p spike-n40 --bin n40-probe -- rebind --n 40 \
  --json crates/spike-N40/results/$(date +%F)-rebind.json
# Optional threshold bracket above 5 events/s if no cascade was observed:
cargo run -p spike-n40 --bin n40-probe -- sweep --n 40 --start 5 --max 80 --factor 2
```

## Privacy

Any JSON emitted by `n40-probe --json` contains only synthetic loopback
measurements (RSS estimates, task-count estimates, accepted / frames_sent
counters, queue-saturation counts, reconnect/sec rates, cascade verdicts) —
no real identities, no network addresses beyond `127.0.0.1`, no message
bodies beyond synthetic `n40 load seq=<k>` labels, no invite / capability
secrets, no blob data, no local filesystem paths outside this results
directory, and no private keys (spec §10).
